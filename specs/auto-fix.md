# Feature: auto-fix

> Generated by `/feature` on 2026-03-28. Review before implementation.

---

## Intent

### 1. What problem does this solve?

Non-technical users build apps with AI tools (Cursor, Claude Code, Bolt, etc.) and the apps are buggy. The user has no idea what's wrong, where to look, or how to describe the problem in terms any debugging tool can act on. The AI that built the app can't reliably fix it because it lacks sufficient context about what's actually broken and what its previous fixes actually did. The result is a "slop cycle" — the AI fixes one thing, breaks another, and the app churns without converging toward working.

Gasoline already has the tools and telemetry to diagnose issues, but the user has to know what to ask for. These users fundamentally can't. What's missing from the market is a guided experience where the AI acts as a QA tester — walks through the entire app, page by page, interaction by interaction — and comes back with a comprehensive, actionable list of everything that's wrong.

### 2. How will we know it worked?

**Primary metric:** Ratio of issues autonomously surfaced and resolved (error disappears from Gasoline's buffers after AI acts on the report) — baselined and measured through manual user testing sessions.

**Supporting signals:**
- Console error count drops to zero after QA + fix cycle
- Screenshots/video show no visible problems post-fix
- App fundamentally runs — no crashes, white screens, or broken core flows
- Number of Gasoline tool calls before the AI finds the actual problem decreases (less flailing)

**Measurement plan:** Manual user testing before and after. Baseline: how many issues does Gasoline catch unprompted vs. how many exist. After: did the ratio improve significantly. Ongoing: instrument console error count pre/post and whether surfaced issues lead to resolution.

### 3. What is out of scope?

- **Feature building / code generation** — this is for fixing and refining existing behavior, not creating new things
- **Being an AI / copilot** — Gasoline stays in the telemetry/recording lane. It does not reason about repair strategy, track fix attempts, or evolve recommendations across iterations
- **UX/design as the primary goal** — visual/design improvements may surface as side effects but are not the mission
- **Stateful fix intelligence** — Gasoline records what happened (action history, replay, reproduction) but does not reason about what to do next. It's a black box recorder, not a copilot

**Explicitly NOT out of scope:** Fix generation is not the goal but is not off the table if it's what's needed to break the slop cycle. The feature may ship guidance that helps the AI generate better fixes.

### 4. What must NOT happen?

1. **Must not provide noisy, ambiguous, or bad data** — quality of diagnosis is Gasoline's responsibility. If Gasoline surfaces garbage, the AI acts on garbage, and the user's app gets worse. Every report must have sufficient context for a competent AI to act on it correctly.
2. **Must not cause side effects in the user's app** — scanning and observation must not trigger destructive actions, submit forms, or alter application state.
3. **Must not try to be an AI** — no stateful fix tracking, no "you already tried that," no intelligence about the repair cycle. Gasoline provides telemetry, evidence, and methodology. The LLM thinks.
4. **Must not break the general-purpose toolkit** — auto-fix is additive. The 5 core tools (observe, analyze, generate, configure, interact) continue to work exactly as they do today for all other use cases. Nothing about auto-fix constrains advanced users.

### 5. Pre-mortem findings

1. **The slop cycle (primary risk):** AI fixes one thing and introduces another. Gasoline faithfully reports the new break. The AI "fixes" that and introduces another. The user watches their app churn without converging. Gasoline did its job perfectly and the experience is still terrible. This is partially outside Gasoline's control — it depends on LLM quality — but the quality of Gasoline's reports directly affects whether the AI can fix correctly the first time.

2. **Extension interference:** On some pages (e.g., Cloudflare), the pages misbehave when Gasoline's extension is active. Reason unknown. Users can't tell if the bug is theirs or Gasoline's. Trust erodes. This is a pre-existing issue, not introduced by auto-fix, but auto-fix makes it more visible because the QA scan touches more pages.

3. **Visibility ceiling:** Some bugs live in places Gasoline can't see — server-side logic, build configuration, environment issues, third-party API behavior. Gasoline reports "everything looks fine" when the app is clearly broken. User loses faith in the tool. The QA report must clearly state what was and wasn't checked.

## Behavioral Spec

### User Stories

**Today (broken experience):**
1. User vibe-codes an app with an AI tool. App has bugs.
2. User installs Gasoline, attaches extension, clicks "track."
3. AI has Gasoline's tools available but no methodology — it calls `observe` once, sees some errors, tries to fix those, and stops.
4. Remaining bugs stay hidden. User doesn't know what to ask for. AI doesn't know to look further.
5. If the user is advanced, they can prompt the AI with detailed debugging instructions. Most users can't.

**Desired experience:**
1. User vibe-codes an app. App has bugs.
2. User installs Gasoline with one-line installer. Everything auto-configures.
3. User opens the side panel terminal (or their AI tool) and says "QA my app" — OR clicks the "Find Problems" button on the hover widget.
4. The AI systematically walks the app: screenshots every page, clicks interactive elements, checks for console errors, network failures, a11y violations, visual issues. It asks the user when something is ambiguous ("is this button supposed to do X?").
5. The AI reports back with a prioritized list of everything wrong, with evidence (screenshots, error logs, DOM state).
6. The AI fixes issues. User clicks around to verify. Done.

### Mechanism

Three delivery channels that work together:

1. **`analyze(what: "page_issues")` — one-call evidence sweep.** Aggregates everything Gasoline can detect without an LLM in a single call: console errors, network failures (4xx/5xx), a11y violations, broken links, form validation issues, security findings. Returns a unified, prioritized report. Platform independent via MCP. This gives ANY AI a comprehensive starting point without needing to know which individual tools to call.

2. **`qa` meta-skill — the methodology playbook.** A skill file installed to `~/.claude/skills/`, `~/.codex/skills/`, `~/.gemini/skills/` that teaches the AI the full QA walkthrough methodology. Covers: initial sweep with `page_issues`, navigation discovery, per-page walkthrough, interactive element testing, evidence gathering, and report synthesis. Works on platforms that consume skill files.

3. **Hover widget "Find Problems" button — the non-technical trigger.** A new button on the floating toolbar that creates a server-side intent with a correlation ID. If the terminal is open, injects a nudge. If not, the intent is surfaced when the AI next calls any Gasoline tool. Both channels carry the same correlation ID — whichever the AI picks up first, the other is discarded as already-handled.

**Why this approach works:**
- `page_issues` is platform-independent (any MCP-connected AI)
- The meta-skill provides methodology for platforms that consume skills
- The hover button provides a zero-knowledge trigger for non-technical users
- The existing toolkit is unchanged — advanced users aren't affected

### States and Transitions

1. **Installed** — Gasoline installed via one-liner. MCP configured. Skills installed. Extension loaded. AI discovers capabilities via MCP server description + installed skill files.

2. **Idle/Tracked** — Extension is tracking a page (green dot in widget). Hover widget visible with "Find Problems" button. Terminal available in side panel. No QA activity.

3. **Intent Created** — User clicks "Find Problems" or says "QA my app." Server-side intent created with correlation ID + timestamp. Terminal gets nudge if open. Transitions to Active when AI picks up the intent.

4. **Active QA** — AI is systematically walking the app using Gasoline tools. Flow:
   - `analyze(what: "page_issues")` for initial sweep
   - `interact(what: "explore_page")` for navigation discovery
   - Per-page: screenshot → page_issues → interact with key elements → gather evidence
   - Ask user about ambiguous behavior

5. **Auth Wall** — Gasoline detects a login page during walkthrough. Pauses. Tells the user "I need you to log in, then I'll continue." User logs in manually. AI resumes.

6. **Report** — AI has completed the walkthrough. Produces prioritized issue list with evidence (screenshots, error logs, DOM state, network traces). User reviews.

7. **Fix Cycle** — AI works through the issue list, fixing each one. Gasoline continues providing telemetry as fixes are applied. User verifies by using the app.

8. **Done** — AI can't find more bugs. User confirms the app works. Console errors at zero. Screenshots show clean state.

### Error Cases

- **AI calls wrong tools / bad parameters:** Existing structured error system handles this — `recovery_playbook`, `hint`, `recovery_tool_call` guide the AI back on track. No change needed.
- **Auth walls:** Gasoline detects login page, pauses for user to handle manually. AI resumes after.
- **Destructive actions during QA:** AI's responsibility to recognize destructive elements. Gasoline surfaces interactive element info (labels, types) — `list_interactive` returns enough context for the AI to avoid "Delete" buttons.
- **Visual bugs (no console errors):** Screenshots + video capture surface these. The AI (being multimodal) can see visual issues in screenshots. `page_issues` won't catch these — the skill methodology must include screenshot review as a step.
- **Extension disconnected mid-scan:** `page_issues` returns partial results with clear indication of which checks were skipped. Existing disconnect warning pattern applies.
- **Page with thousands of errors:** `page_issues` caps results per category and includes total counts. Pagination via existing cursor pattern.
- **Stale intent:** Server-side intents need a TTL (e.g., 5 minutes). Expired intents are discarded, not surfaced.

## Design Approach

### Conceptual approach

The feature is delivered through three independent, composable pieces rather than a single monolithic change:

1. A new analyze mode (`page_issues`) that aggregates existing detection capabilities into one call — giving any AI a comprehensive starting point without needing to know Gasoline's tool taxonomy.
2. A meta-skill (`qa`) that teaches the AI the full walkthrough methodology — bridging the gap between "tools exist" and "AI knows how to use them systematically."
3. A hover widget button that creates a server-side intent — bridging the gap between "user knows something is wrong" and "AI starts working on it."

**Alternatives considered:**
- **6th MCP tool (`qa`):** Rejected. QA requires an LLM to reason about what it sees. Gasoline can assemble evidence but can't drive the walkthrough server-side. The methodology must live in the AI layer.
- **Orchestration mode (Gasoline drives exploration):** Rejected for the same reason. Gasoline would need to decide what to click, what's a bug, what to investigate further. That's LLM work.
- **Enhanced tool responses only (no skill):** Insufficient. Tool response hints help but can't convey a full multi-step methodology. The skill file is the right vehicle for teaching a workflow.

### Codebase context

**Analyze dispatch** (`tools_analyze_dispatch.go`): Clean registry pattern. New modes are added by creating a handler function and registering it in the `analyzeHandlers` map. Three handler patterns exist: local method, observe package delegation, and security scanner delegation. `page_issues` will follow the security_audit pattern (aggregates from multiple buffers).

**Skill system** (`npm/gasoline-agentic-browser/skills/`): 14 existing skills with clean structure. Each skill is a standalone `SKILL.md` with YAML frontmatter, workflow steps, and output contract. Skills don't reference each other — they reference tools. The `qa` skill will use the same tools that `site-audit`, `debug-triage`, and `ux-audit` use, but orchestrate them into a single methodology.

**Hover widget** (`tracked-hover-launcher.ts`): Buttons are created via `createActionButton()` and dispatch via `chrome.runtime.sendMessage()`. Annotation flow already demonstrates the pattern of injecting context into the terminal. Adding a "Find Problems" button follows the exact same pattern as the screenshot button.

**Existing relevant skills:** `site-audit` (comprehensive 12-step product audit), `debug-triage` (single-bug root cause), `ux-audit` (UX quality review). The `qa` meta-skill will synthesize these into one entry point with a focus on the non-technical user persona.

## Implementation Design

### Architecture

**Piece 1: `analyze(what: "page_issues")`**

| Aspect | Detail |
|--------|--------|
| File | New: `cmd/browser-agent/tools_analyze_page_issues.go` |
| Registration | Add to `analyzeHandlers` map in `tools_analyze_dispatch.go` |
| Pattern | Follows `security_audit` model: aggregate from multiple buffers, run checks, build summary |
| Handler signature | `func (h *ToolHandler) toolAnalyzePageIssues(req JSONRPCRequest, args json.RawMessage) JSONRPCResponse` |

**Data sources aggregated:**
- Console errors: `h.capture.GetLogEntries()` — filter for errors/warnings
- Network failures: `h.capture.GetNetworkWaterfallEntries()` — filter 4xx/5xx
- A11y violations: `h.ExecuteA11yQuery()` — extension async query
- Security findings: `h.securityScannerImpl.HandleSecurityAudit()` — from existing scanner
- Form issues: extension query for `form_validation`
- Link health: extension query for broken links

**Fan-out strategy:** Run extension queries (a11y, forms, links) in parallel using goroutines. Collect results with a timeout per check (e.g., 5s each, 20s total). Return partial results if any check times out, with clear indication of which checks completed vs. skipped.

**Response structure:**
```json
{
  "total_issues": 12,
  "by_severity": {"critical": 2, "high": 3, "medium": 5, "low": 2},
  "sections": {
    "console_errors": [...],
    "network_failures": [...],
    "accessibility_violations": [...],
    "security_findings": [...],
    "form_issues": [...],
    "broken_links": [...]
  },
  "checks_completed": ["console", "network", "accessibility", "security"],
  "checks_skipped": ["forms"],
  "page_url": "http://localhost:3000",
  "timestamp": "2026-03-28T..."
}
```

Supports `summary: true` for compact output (~80% reduction).

**Piece 2: `qa` meta-skill**

| Aspect | Detail |
|--------|--------|
| Directory | `npm/gasoline-agentic-browser/skills/qa/SKILL.md` |
| Manifest entry | Add to `skills.json` with id `"qa"`, description targeting "QA my app" / "find all bugs" / "comprehensive quality review" triggers |
| Install targets | `~/.claude/skills/qa.md`, `~/.codex/skills/qa/SKILL.md`, `~/.gemini/skills/qa.md` via existing `install-bundled-skills.sh` |

**Methodology the skill teaches:**
1. Run `analyze(what: "page_issues", summary: true)` for initial sweep
2. Take screenshot of current page, assess visual state
3. Discover navigation: `interact(what: "explore_page")` + `interact(what: "list_interactive")`
4. Build page queue from navigation links, sidebar, footer, CTAs
5. For each page: navigate → screenshot → `page_issues` → interact with key elements → note issues
6. If auth wall detected: pause, ask user to log in, resume
7. For each issue found: gather targeted evidence (error_bundles, DOM state, network_bodies)
8. Ask user about ambiguous behavior ("is this button supposed to do X?")
9. Produce prioritized report: critical → high → medium → low, with evidence per issue
10. Offer to fix issues, starting with highest severity

**Piece 3: Hover widget button + server-side intent**

| Aspect | Detail |
|--------|--------|
| Button code | `tracked-hover-launcher.ts` — new `createActionButton()` call |
| Button icon | TBD (magnifying glass, bug icon, or shield) |
| Message type | `'qa_scan_requested'` in `runtime-messages.ts` |
| Server handler | New intent storage in Go daemon with correlation ID + TTL |
| Intent surface | `observe(what: "pending_intents")` and/or injected as metadata in next tool response |
| Terminal nudge | If terminal visible, inject: "Gasoline QA scan requested — correlation_id: {id}" |
| Dedup | Correlation ID carried in both channels. AI picks up first one, second is discarded as handled. |

**Intent lifecycle:**
1. Button click → generate correlation ID → `chrome.runtime.sendMessage({ type: 'qa_scan_requested', correlation_id })`
2. Background service worker forwards to daemon via existing WebSocket/HTTP channel
3. Daemon stores intent with correlation ID + timestamp + 5-minute TTL
4. If terminal open: inject nudge with same correlation ID
5. AI picks up via terminal nudge OR via `observe(what: "pending_intents")` OR via metadata in next tool response
6. Once consumed (any channel), intent is marked handled. Same correlation ID from other channel is discarded.
7. After TTL, unconsumed intents are silently dropped.

### Structural issues flagged

**No major structural issues.** The codebase has clean extension patterns for all three pieces:
- Analyze dispatch is a simple registry — adding a mode is mechanical
- Skill system is file-based — adding a skill is creating a markdown file
- Hover widget has a clear button factory pattern

**One new concern: fan-out in `page_issues`.** The existing async pattern handles single extension queries. `page_issues` needs to fan out 3-4 extension queries in parallel and aggregate results. This requires new goroutine coordination in the Go handler. Not complex, but it's new plumbing that doesn't exist yet. Use `sync.WaitGroup` or channel-based fan-out with per-check timeouts.

### Constraints and Non-negotiables

- **Security:** No new attack surface. `page_issues` reads existing buffers and runs existing queries. No new permissions. No new network calls. No data leaves localhost.
- **Reliability:** `page_issues` must return partial results on timeout, never fail entirely. If 3 of 5 checks complete and 2 timeout, return the 3 with clear `checks_skipped` field. Total timeout budget: 20 seconds.
- **Performance:** WebSocket < 0.1ms, HTTP < 0.5ms (existing budgets). `page_issues` total response time < 20s including all extension queries.
- **Zero deps:** Go side stays zero production dependencies. Skill is a markdown file. Button is vanilla JS in the extension.
- **Wire types:** If `page_issues` introduces new response types, update `wire_*.go` and `wire-*.ts` counterparts. Run `make check-wire-drift`.
- **File size:** Handler must stay under 800 LOC. If fan-out logic is complex, extract to `internal/tools/observe/` or a new `internal/aggregate/` package.

### Known AI Blind Spots

- **[x] Edge cases:** Page with zero issues must return a clean report (not empty/null). Page with thousands of errors needs per-section caps (e.g., 50 per category) with total counts. Empty page (about:blank) should return immediately with "no content to analyze." Pages behind CSP that block extension scripts need graceful degradation.
- **[x] Concurrency:** Fan-out goroutines for parallel extension queries. Must handle: one query completes while another is still pending; all queries timeout; extension disconnects mid-fan-out. Use per-check channels with select + timeout, not unbounded goroutines.
- **[x] Error handling:** Extension disconnected → return partial results with `checks_skipped`. Daemon not running → existing recovery pattern (structured error with `recovery_tool_call` pointing to `configure(what: "restart")`). Individual check fails → skip that check, include in `checks_skipped`, don't fail the whole call.
- **[x] Security:** No new attack surface. All data stays local. Intent storage must not be accessible cross-origin. Correlation IDs must not be guessable (use crypto random, not sequential).
- **[x] Non-prompted concerns:** Hover button needs debounce (prevent rapid double-clicks creating multiple intents). Server-side intents need TTL (5 minutes) and max pending count (e.g., 3) to prevent intent accumulation. `page_issues` response size could be large — respect existing `ClampResponseSize()` pipeline. Skill file must ship in the same release as `page_issues` mode or the skill will reference a mode that doesn't exist.
- **[x] Hallucination risk:** The `qa` skill references `analyze(what: "page_issues")` — must ship together. All tool calls in the skill must reference real, existing modes. Review every `observe`/`analyze`/`interact`/`generate` call in the skill against the actual registries before shipping. No invented parameters.

### Rollback Plan

- **Trigger signal:** `page_issues` returns incorrect/noisy results that cause AIs to take wrong actions; hover button creates confusing behavior; skill methodology produces poor QA results in user testing.
- **Method:** All three pieces are independently reversible:
  - `page_issues`: Remove from `analyzeHandlers` registry. One line change.
  - `qa` skill: Delete skill directory + remove from `skills.json`. File deletion.
  - Hover button: Remove button creation from `tracked-hover-launcher.ts`. One code block removal.
  - Server-side intents: Remove handler. No persistent state affected.
