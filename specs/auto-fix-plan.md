### Phase 1: `analyze(what: "page_issues")` — Go handler + schema
**Files:**
- New: `cmd/browser-agent/tools_analyze_page_issues.go` (~120 LOC)
- New: `cmd/browser-agent/tools_analyze_page_issues_summary.go` (~80 LOC)
- New: `cmd/browser-agent/tools_analyze_page_issues_test.go`
- Edit: `cmd/browser-agent/tools_analyze_dispatch.go` (1 line — add to registry)
- Edit: `internal/schema/analyze.go` (1 line — add to enum)

**Changes:**
Handler that aggregates from existing data sources in a single call:
- Console errors from `h.capture.GetLogEntries()` — filter for error/warning level
- Network failures from `h.capture.GetNetworkWaterfallEntries()` — filter 4xx/5xx status
- A11y violations via `h.ExecuteA11yQuery()` — extension async query
- Security findings via `h.securityScannerImpl.HandleSecurityAudit()` — existing scanner
- Form issues via extension query for `form_validation`

Fan-out strategy: run extension queries (a11y, forms) in parallel using goroutines with per-check timeouts (5s each, 20s total). Return partial results if any check times out with `checks_completed`/`checks_skipped` fields. Follows the `security_audit` handler pattern (aggregate from multiple buffers, delegate to internal package, build summary).

Response structure:
```json
{
  "total_issues": 12,
  "by_severity": {"critical": 2, "high": 3, "medium": 5, "low": 2},
  "sections": {
    "console_errors": [...],
    "network_failures": [...],
    "accessibility_violations": [...],
    "security_findings": [...],
    "form_issues": [...]
  },
  "checks_completed": ["console", "network", "accessibility", "security"],
  "checks_skipped": ["forms"],
  "page_url": "http://localhost:3000",
  "timestamp": "2026-03-28T..."
}
```

Supports `summary: true` for compact output (~80% reduction).

### Phase 2: `qa` meta-skill + skills.json
**Files:**
- New: `npm/gasoline-agentic-browser/skills/qa/SKILL.md` (~100 LOC)

- Edit: `pypi/gasoline-agentic-browser/gasoline_agentic_browser/skills/skills.json` (add qa entry)

**Changes:**
Meta-skill that teaches any AI the full QA walkthrough methodology. Structure follows site-audit pattern (comprehensive workflow with checklists) but focused on the non-technical user persona and the "QA my app" trigger.

Skill methodology (10 steps):
1. Run `analyze(what: "page_issues", summary: true)` for initial evidence sweep
2. Take screenshot of current page, assess visual state
3. Discover navigation: `interact(what: "explore_page")` + `interact(what: "list_interactive")`
4. Build page queue from navigation links, sidebar, footer, CTAs
5. For each page: navigate → screenshot → `page_issues` → interact with key elements → note issues
6. If auth wall detected: pause, ask user to log in, resume
7. For each issue found: gather targeted evidence (error_bundles, DOM state, network_bodies)
8. Ask user about ambiguous behavior ("is this button supposed to do X?")
9. Produce prioritized report: critical → high → medium → low, with evidence per issue
10. Offer to fix issues, starting with highest severity

Output contract: `issue_inventory`, `page_coverage`, `evidence_map`, `priority_backlog`, `checks_summary`

skills.json entry:
```json
{
  "id": "qa",
  "version": 1,
  "description": "Use when user says 'QA my app', 'find all bugs', 'check my app for problems', 'debug my whole app', or wants a comprehensive quality review of their web application."
}
```

**Changes:**

**TypeScript (extension):**
- Add `QaScanRequestedMessage` interface to `runtime-messages.ts` with `type: 'qa_scan_requested'`, optional `correlation_id: string`, optional `page_url: string`
- Add to `BackgroundMessage` discriminated union
- Add "Find Problems" button to hover widget using `createActionButton()` pattern. Icon: magnifying glass or bug. Position: before terminal button.
- Click handler: generate correlation ID via `crypto.randomUUID()`, send `qa_scan_requested` runtime message, debounce 500ms to prevent double-clicks
- If terminal visible (`isTerminalVisible()`): inject nudge via `writeToTerminal()` with correlation ID — "Gasoline detected issues on this page. Run the QA skill or call analyze(what: 'page_issues') to investigate. Correlation ID: {id}"
- Add handler in `message-handlers.ts` switch: forward to daemon via existing HTTP/WebSocket channel

**Go (daemon):**
- `intent_store.go`: In-memory store with:
  - `AddIntent(correlationID, pageURL, tabID)` — stores with timestamp
  - `ConsumeIntent(correlationID)` — returns and removes intent
  - `GetPendingIntents()` — returns all non-expired intents
  - `cleanExpired()` — removes intents older than 5 minutes TTL
  - Max 3 pending intents (oldest evicted on overflow)
  - Correlation IDs from `crypto/rand` (not sequential)
- Post-dispatch hook or `tools_core.go` edit: after any tool call succeeds, check intent store. If pending intents exist, append to response metadata: `"pending_intents": [{"correlation_id": "...", "page_url": "...", "created_at": "..."}]`
- HTTP endpoint to receive intent from extension: `POST /intent` with correlation_id + page_url
