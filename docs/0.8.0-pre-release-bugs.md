### BUG: `clearAllEventBuffers()` does not reset monotonic counters

**Files:** `internal/capture/buffer_store.go:102-111`, called from `internal/capture/buffer_clear.go:94`

The `BufferStore` maintains per-buffer monotonic counters that track cumulative totals across the lifetime of the daemon. These counters are exposed via the `/health` endpoint and used by LLM clients to assess buffer activity.

Each individual clear method correctly resets its counters:
 
```go
// buffer_store.go:81-100
func (s *BufferStore) clearNetworkBuffers() {
    s.networkBodies = make([]NetworkBody, 0)
    s.networkAddedAt = make([]time.Time, 0)
    s.networkTotalAdded = 0         // <-- counter reset
    s.networkErrorTotalAdded = 0    // <-- counter reset
    s.networkBodyMemoryTotal = 0
}

func (s *BufferStore) clearWebSocketBuffers() {
    s.wsEvents = make([]WebSocketEvent, 0)
    s.wsAddedAt = make([]time.Time, 0)
    s.wsTotalAdded = 0              // <-- counter reset
    s.wsMemoryTotal = 0
}

func (s *BufferStore) clearActionBuffers() {
    s.enhancedActions = make([]EnhancedAction, 0)
    s.actionAddedAt = make([]time.Time, 0)
    s.actionTotalAdded = 0          // <-- counter reset
}
```

But `clearAllEventBuffers()` — the method called by `ClearAll()` — only resets slices and memory totals, not the four monotonic counters:

```go
// buffer_store.go:102-111
func (s *BufferStore) clearAllEventBuffers() {
    s.wsEvents = make([]WebSocketEvent, 0)
    s.wsAddedAt = make([]time.Time, 0)
    s.wsMemoryTotal = 0
    s.networkBodies = make([]NetworkBody, 0)
    s.networkAddedAt = make([]time.Time, 0)
    s.networkBodyMemoryTotal = 0
    s.enhancedActions = make([]EnhancedAction, 0)
    s.actionAddedAt = make([]time.Time, 0)
    // MISSING: wsTotalAdded, networkTotalAdded, networkErrorTotalAdded, actionTotalAdded
}
```

`ClearAll()` is the public entry point at `buffer_clear.go:90-101`:

```go
func (c *Capture) ClearAll() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.buffers.clearAllEventBuffers()  // <-- calls the broken method
    c.networkWaterfall.clear()
    c.wsConnections.clear()
    c.extensionState.activeTestIDs = make(map[string]bool)
    c.perf.clear()
}
```

**Impact:** After `configure({what:"clear", buffer:"all"})`, the `/health` endpoint reports inflated cumulative totals while all buffers are empty. An LLM client reading health data sees, for example, `wsTotalAdded: 347` with zero actual entries — a misleading state that could trigger incorrect diagnostic logic. Meanwhile, `configure({what:"clear", buffer:"network"})` correctly resets counters because it calls `clearNetworkBuffers()` directly.

**Fix:** Replace the body of `clearAllEventBuffers()` with delegation to the three per-buffer methods. This is both DRY and automatically fixes the counter bug:

```go
func (s *BufferStore) clearAllEventBuffers() {
    s.clearNetworkBuffers()
    s.clearWebSocketBuffers()
    s.clearActionBuffers()
}
```

### BUG: Missing `response.ok` checks on 2 fetch calls

The CLAUDE.md rule is explicit: "All fetch() needs try/catch + response.ok check." Two calls violate this.

**Location 1:** `src/background/recording-listeners.ts:213-220`

```typescript
fetch(`${deps.getServerUrl()}/recordings/reveal`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json', 'X-Gasoline-Client': 'gasoline-extension' },
  body: JSON.stringify({ path: message.path })
})
  .then((r) => r.json())       // <-- attempts .json() on 4xx/5xx responses
  .then((result) => sendResponse(result))
  .catch((err) => sendResponse({ error: (err as Error).message }))
```

The `fetch()` API does **not** throw on HTTP error status codes (4xx/5xx). A 404 or 500 response is considered a successful network request. The `.then((r) => r.json())` call will attempt to parse the error body as JSON — if the server returns an HTML error page, this silently fails and the promise chain either returns a malformed object or falls into the `.catch()` with a confusing JSON parse error instead of the actual HTTP error.

**Location 2:** `src/background/index.ts:305-313`

```typescript
export async function handleClearLogs(): Promise<{ success: boolean; error?: string }> {
  try {
    await fetch(`${getServerUrl()}/logs`, { method: 'DELETE', headers: getRequestHeaders() })
    setConnectionStatus({ entries: 0, errorCount: 0 })  // <-- assumes success
    updateBadge(getConnectionStatus())
    return { success: true }                              // <-- always reports success
  } catch (error) {
    return { success: false, error: (error as Error).message }
  }
}
```

The response is completely ignored. A 500 Internal Server Error on the DELETE endpoint is treated as a successful log clear. The badge and connection status are reset even though the server didn't actually delete anything.

Most other fetch calls in the codebase correctly check `response.ok`. One additional call in `src/background/upload-handler.ts:228` (`dismissFileDialog`) also skips `response.ok`, but this is intentionally best-effort (documented as "Best-effort cleanup — ignore errors"). Similarly, `src/lib/link-health.ts:143` uses `mode: 'no-cors'` which produces an opaque response where `.ok` is always false by design.

**Fix:** Add `response.ok` checks:

```typescript
// Location 1:
.then((r) => {
  if (!r.ok) throw new Error(`recordings/reveal failed: ${r.status}`)
  return r.json()
})

// Location 2:
const resp = await fetch(`${getServerUrl()}/logs`, { method: 'DELETE', headers: getRequestHeaders() })
if (!resp.ok) return { success: false, error: `Clear logs failed: ${resp.status}` }
```

**Files:**

- `src/background/commands/analyze.ts:283-286`

```typescript
function isContentScriptUnreachableError(err: unknown): boolean {
```

### DRY: `ai_web_pilot_disabled` guard pattern — 7 inconsistent copies

**File:** `src/background/commands/interact.ts` (lines 110, 130, 151, 172, 210, 236, 354)

Seven command handlers check whether AI Web Pilot (the browser automation feature) is enabled before proceeding. The pattern is structurally identical but subtly inconsistent:

```typescript
// Pattern at lines 110-122 (dom_action) — passes null:
if (!isAiWebPilotEnabled()) {
  if (ctx.query.correlation_id)
    ctx.sendAsyncResult(ctx.syncClient, ctx.query.id, ctx.query.correlation_id, 'error', null, 'ai_web_pilot_disabled')
  else ctx.sendResult({ error: 'ai_web_pilot_disabled' })
  return
}

// Pattern at lines 172-185 (record_start) — passes undefined:
if (!isAiWebPilotEnabled()) {
  if (ctx.query.correlation_id)
    ctx.sendAsyncResult(ctx.syncClient, ctx.query.id, ctx.query.correlation_id, 'error', undefined, 'ai_web_pilot_disabled')
  else ctx.sendResult({ error: 'ai_web_pilot_disabled' })
  return
}

// Pattern at lines 236-239 (state_*) — simpler, no correlation_id handling:
if (!isAiWebPilotEnabled()) {
  ctx.sendResult({ error: 'ai_web_pilot_disabled' })
  return
}
```

The inconsistencies:
1. `null` vs `undefined` as the result parameter in `sendAsyncResult` — different serialization in JSON (`null` vs omitted)
2. Some commands route through `correlation_id` handling; `state_*` does not
3. None include a human-readable `message` field explaining the error

**Impact:** An LLM client receives different error shapes for the same condition depending on which command it called. The `state_*` handler silently drops the async path, so an async caller never receives a response.

**Fix:** Extract to `helpers.ts`:

```typescript
export function requireAiWebPilot(ctx: CommandContext): boolean {
  if (isAiWebPilotEnabled()) return true
  if (ctx.query.correlation_id)
    ctx.sendAsyncResult(ctx.syncClient, ctx.query.id, ctx.query.correlation_id, 'error', null, 'ai_web_pilot_disabled')
  else ctx.sendResult({ error: 'ai_web_pilot_disabled', message: 'AI Web Pilot is not enabled in the extension popup' })
  return false
}
```

Each handler becomes a one-liner: `if (!requireAiWebPilot(ctx)) return`.

### DRY: Manual sync/async response routing — 24 redundant copies

**Files:** `analyze.ts` (6 occurrences), `observe.ts` (2), `interact.ts` (8), `interact-explore.ts` (2), `analyze-navigation.ts` (2), `analyze-feature-gates.ts` (2), `analyze-page-structure.ts` (2)

All 24 sites manually check `ctx.query.correlation_id` to decide whether to send a synchronous or asynchronous result:

```typescript
// Example from interact.ts:201-203 (record_start handler):
if (ctx.query.correlation_id)
  ctx.sendAsyncResult(ctx.syncClient, ctx.query.id, ctx.query.correlation_id, statusFromError(error), result, error)
else ctx.sendResult(error ? { error } : result)
```

The command registry's `createDispatchLifecycle` in `registry.ts:95-183` was explicitly designed to handle this routing. The lifecycle wraps each handler with target resolution, init-ready gating, and response routing. `ctx.sendResult` and `ctx.sendAsyncResult` are already provided on the context — but `ctx.sendResult` should handle the routing decision internally rather than requiring each handler to implement it.

**Impact:** Every new command handler must manually implement the routing decision. If the routing logic changes (e.g., adding a new async response format), all 24 sites must be updated. Some handlers route differently from others, creating inconsistent behavior for the same async call pattern.

**Fix:** Handlers should call `ctx.sendResult(result)` unconditionally. The lifecycle should detect `correlation_id` and route automatically. This would require updating the `sendResult` implementation in the dispatch lifecycle to check for `correlation_id` before sending. All 24 manual blocks can then be simplified to a single `ctx.sendResult(result)` call.

### DRY: Redundant params re-parsing — 7 copies bypass existing infrastructure

**Files:** `src/background/commands/interact.ts` (lines 35, 55, 88, 188, 243, 378), `src/background/commands/analyze.ts:659`

The command registry already parses parameters into `ctx.params` via `parseQueryParamsObject()` (defined in `helpers.ts:206`). But seven handlers manually re-parse `ctx.query.params`:

```typescript
// Example from interact.ts:186-191 (record_start handler):
let params: { name?: string; fps?: number; audio?: string }
try {

}
```

This is exactly what `parseQueryParamsObject()` already does. The registry calls it and stores the result in `ctx.params`. Each of these 7 blocks is 5-6 lines of dead-weight boilerplate that duplicates existing infrastructure.

**Fix:** Replace all 7 blocks with `const params = ctx.params as SomeType`. The params are already parsed.

### TYPE: `LogEntry` type fragmented across 5 packages

The canonical definition lives in `internal/types/log.go:20`:

```go
// LogEntry represents a generic log entry from the server.
type LogEntry map[string]any
```

But three packages independently redefine it instead of aliasing:

```go
// internal/mcp/types.go:84-86 — independent redefinition:
// LogEntry represents a single log entry from the browser console.
type LogEntry = map[string]any

// internal/security/security_types.go:24 — independent redefinition:
type LogEntry map[string]any

// internal/pagination/pagination_logs.go:9-11 — independent redefinition:
// LogEntry is a map-based log entry from the capture package.
type LogEntry = map[string]any
```

And `noise` chains through `server` instead of aliasing `types` directly:

```go
// internal/noise/noise.go:17-18
type LogEntry = server.LogEntry  // server.LogEntry = types.LogEntry
```

Go's structural typing makes `map[string]any` compatible everywhere. The `=` alias and the direct definition are both `map[string]any` and interchangeable. But this is fragile:

**Impact:** If `types.LogEntry` is ever changed to a struct (e.g., for performance, field validation, or adding methods), the three independent `map[string]any` definitions will silently diverge. Code passing a `security.LogEntry` to a function expecting `types.LogEntry` would fail to compile only if they diverge — until then, there is no compiler enforcement that they represent the same concept.

**Fix:** Replace all independent definitions with the alias form pointing at the canonical source:

```go
// In mcp/types.go, security/security_types.go, pagination/pagination_logs.go:
type LogEntry = types.LogEntry

// In noise/noise.go (cut the chain through server):
type LogEntry = types.LogEntry
```

### PERF: Bubble sort in production code — O(n^2), duplicated

**File:** `cmd/dev-console/tools_analyze_security.go:119-128, 159-168`

Two identical bubble-sort implementations sort security findings by severity:

```go
// Lines 119-128 (security audit summary):
sorted := make([]security.Finding, len(result.Findings))
copy(sorted, result.Findings)
for i := 0; i < len(sorted)-1; i++ {
    for j := i + 1; j < len(sorted); j++ {
        if severityOrder[sorted[j].Severity] < severityOrder[sorted[i].Severity] {
            sorted[i], sorted[j] = sorted[j], sorted[i]
        }
    }
}

// Lines 159-168 (third-party audit summary) — structurally identical:
sorted := make([]analysis.ThirdPartyEntry, len(result.ThirdParties))
copy(sorted, result.ThirdParties)
for i := 0; i < len(sorted)-1; i++ {
    for j := i + 1; j < len(sorted); j++ {
        if severityOrder[sorted[j].RiskLevel] < severityOrder[sorted[i].RiskLevel] {
            sorted[i], sorted[j] = sorted[j], sorted[i]
        }
    }
}
```

This is selection sort (O(n^2)), not bubble sort, but the complexity issue is the same. While the data sets are typically small (<100 findings), Go's standard library provides `sort.Slice` which is O(n log n) and far more readable. The duplication means any fix must be applied in two places.

**Fix:** Use `sort.Slice`:

```go
sort.Slice(sorted, func(i, j int) bool {
    return severityOrder[sorted[i].Severity] < severityOrder[sorted[j].Severity]
})
```

If both call sites need the same pattern, extract `sortBySeverity[T any](items []T, key func(T) string, order map[string]int)`.

### DEAD CODE: Custom `min`/`max` shadow Go 1.24 builtins

**File:** `internal/tools/analyze/image_diff_math.go:23-35`

```go
func min(a, b int) int {
    if a < b {
        return a
    }
    return b
}

func max(a, b int) int {
    if a > b {
        return a
    }
    return b
}
```

The project uses `go 1.24.13` (verified in `go.mod`). Go has included builtin `min` and `max` functions since Go 1.21 — they work for all ordered types (int, float64, string, etc.) and are generic. These custom implementations shadow the builtins for `int` only, which means:

1. They are functionally identical to the builtins (same behavior for `int`)
2. They hide the builtins, so if someone tries to use `min(a, b float64)` in this package, it won't compile
3. They add cognitive overhead for any reader who wonders whether they differ from the builtins

**Fix:** Delete both functions. The 2 call sites in `image_diff_math.go` (and the `repairActionParallelArrays` call in `buffer_store.go:119` which also uses `min`) will automatically resolve to the builtin.

### SECURITY: Mutable exported globals for SSRF protection

**File:** `internal/upload/ssrf.go:17-23`

```go
// SSRFAllowedHostsList holds host or host:port values that bypass SSRF checks.
// Set via --ssrf-allow-host flag (repeatable). Intended for test use only.
var SSRFAllowedHostsList []string

// SkipSSRFCheck disables private IP blocking in tests where httptest.NewServer
// uses 127.0.0.1. Must only be set from test code.
var SkipSSRFCheck bool
```

These are **exported, mutable, unsynchronized** package-level variables that control security-critical behavior (SSRF protection). Problems:

1. **Data race:** Concurrent goroutines can read `SkipSSRFCheck` while a test sets it — this is a race condition under `-race`
2. **Test isolation:** Test code that sets `SkipSSRFCheck = true` affects all other tests running in the same process in parallel
3. **The comment says "test use only"** but the variable is exported and settable from any package including production code

The `SkipSSRFCheck` variable is read inside the SSRF dial function which runs per-upload-request on the hot path.

**Fix:** Either:
1. Protect with `sync.RWMutex` and use `t.Cleanup()` in tests to restore original values, or
2. Refactor to a config struct injected via function parameter: `func NewSSRFDialer(config SSRFConfig) *net.Dialer`

Option 2 is cleaner and eliminates the global entirely.

### TYPE: `CSPGenerator` interface is empty — zero compile-time safety

**File:** `internal/capture/interfaces.go:15-22`

### DRY: `parseArgs[T]` generic helper exists but 62 handlers parse manually

**File:** `cmd/dev-console/tools_validation.go:18-28`

```go
// parseArgs unmarshals JSON args into a typed struct, returning a structured error on failure.
// Usage: params, resp, ok := parseArgs[MyParams](req, args); if !ok { return resp }
func parseArgs[T any](req JSONRPCRequest, args json.RawMessage) (T, JSONRPCResponse, bool) {
    var p T
    if err := json.Unmarshal(args, &p); err != nil {
        return p, JSONRPCResponse{JSONRPC: "2.0", ID: req.ID, Result: mcpStructuredError(
            ErrInvalidJSON, "Invalid JSON arguments: "+err.Error(), "Fix JSON syntax and call again",
        )}, false
    }
    return p, JSONRPCResponse{}, true
}
```

This is a well-designed generic helper that eliminates the manual unmarshal+error pattern. But across `cmd/dev-console/`, **62 handlers** still use the manual pattern:

```go
// Repeated 62 times across tool handlers:
var params struct { /* fields */ }
if len(args) > 0 {
    if err := json.Unmarshal(args, &params); err != nil {
        return JSONRPCResponse{JSONRPC: "2.0", ID: req.ID, Result: mcpStructuredError(
            ErrInvalidJSON, "Invalid JSON arguments: "+err.Error(), "Fix JSON syntax and call again")}
    }
}
```

### ARCH: Parallel arrays in `BufferStore` are structurally fragile

**File:** `internal/capture/buffer_store.go`

The `BufferStore` maintains three pairs of parallel slices that must be kept in sync:

```
wsEvents       []WebSocketEvent    ←→  wsAddedAt      []time.Time
networkBodies  []NetworkBody       ←→  networkAddedAt []time.Time
enhancedActions []EnhancedAction   ←→  actionAddedAt  []time.Time
```

The `append` methods (e.g., `appendEnhancedActions` at line 124) add to both slices together, and the eviction methods remove from both. But if a bug causes them to desynchronize, three `repair*ParallelArrays()` methods exist as defensive recovery:

```go
// buffer_store.go:113-122
func (s *BufferStore) repairActionParallelArrays() {
    if len(s.enhancedActions) == len(s.actionAddedAt) {
        return
    }
    fmt.Fprintf(os.Stderr, "[gasoline] WARNING: enhancedActions/actionAddedAt length mismatch: %d != %d (recovering by truncating)\n",
        len(s.enhancedActions), len(s.actionAddedAt))
    minLen := min(len(s.enhancedActions), len(s.actionAddedAt))
    s.enhancedActions = s.enhancedActions[:minLen]
    s.actionAddedAt = s.actionAddedAt[:minLen]
}
```

The existence of `repair*` methods means this class of bug has happened before. The `test_helpers.go` file even has `SetWSParallelMismatchForTest` to deliberately create mismatches for testing the repair logic.

**Impact:** If arrays go out of sync, the repair silently truncates data with only a stderr warning. Pagination cursors that reference positions in the old array become invalid. Timestamps get associated with the wrong events.

**Fix:** Replace parallel arrays with a single slice of a struct:

```go
type timestampedEvent[T any] struct {
    event   T
    addedAt time.Time
}

// In BufferStore:
wsEvents        []timestampedEvent[WebSocketEvent]
networkBodies   []timestampedEvent[NetworkBody]
enhancedActions []timestampedEvent[EnhancedAction]
```

This makes length mismatches structurally impossible. Delete the three `repair*` methods. The existing `RingBuffer[T]` in `internal/buffers/` already implements this pattern and could potentially be reused.
