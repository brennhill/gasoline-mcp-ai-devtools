## New Findings

### HIGH Priority

#### H1. Integration tests do not compile (pre-existing, now exposed by `test-integration` target)

All 15 `//go:build integration` files in `internal/` reference types from `cmd/dev-console/` (`JSONRPCRequest`, `MCPTool`, `setupToolHandler`, etc.) that are not accessible from internal packages. `make test-integration` will fail for `internal/...`.

**Affected packages:** `internal/analysis`, `internal/queries`, `internal/security`, `internal/export`, `internal/performance`, `internal/redaction`, `internal/audit`

**This was always broken** — the Makefile just didn't have the target before. Now that it does, this is exposed.

**Fix options:**
1. Move these tests to `cmd/dev-console/` where the types live
2. Create exported test helpers in a `testutil` package
3. Restructure type dependencies

#### H2. `SSRFAllowedHosts()` returns shared slice reference, not a defensive copy

**File:** `internal/upload/ssrf.go:32-37`

Comment says "returns a snapshot" but returns the underlying slice directly. Safe today (SetSSRFAllowedHosts replaces the whole slice), but fragile API that invites future data races.

#### H3. `parseArgs[T]` generic helper is unused (dead code in production)

**File:** `cmd/dev-console/tools_validation.go:20-28`

Created as infrastructure for 62 handlers but never adopted. Either adopt it or remove it.

#### H4. Two `debugLog` functions with same name but different behavior (TS)

**Files:** `src/background/commands/helpers.ts:67` vs `src/background/index.ts:93`

- `helpers.ts` version: lightweight, `console.debug` only, NOT persisted
- `index.ts` version: full logging, persisted to daemon via `addDebugLogEntry`

Registry-level diagnostics are silently lost because they use the non-persisting version.

### MEDIUM Priority

#### M1. 182 test file headers have incorrect `// Docs:` links

The automated header agent applied a blanket default (`docs/features/feature/observe/index.md`) to ~150 test files regardless of their actual domain. `circuit/breaker_test.go`, `buffers/ring_buffer_test.go`, `recording/manager_test.go` etc. all point to the observe feature docs.

#### M2. Image diff grid logic duplicated in two files

**Files:** `internal/tools/analyze/image_diff.go:42-75` and `image_diff_grid.go:7-45`

`CompareImages` builds a `changed [][]bool` grid inline, duplicating `RebuildChangedGrid`. Future comparison algorithm changes would need updating in both places.

#### M3. `WriteDiffImagePublic` naming anti-pattern

**File:** `internal/tools/analyze/image_diff_render.go:12-14`

Public wrapper around private function with "Public" in the name. Go anti-pattern.

#### M4. Sync/async response routing still duplicated ~20 times (TS)

The `if (ctx.query.correlation_id) { sendAsyncResult } else { sendResult }` pattern appears in ~20 command handlers. The registry lifecycle already handles this automatically for handlers that use `ctx.sendResult()` only.

#### M5. Three differently-named wrappers for `util.ExtractOrigin`

**Files:** `security/security_url_helpers.go` (`extractOrigin`), `security/csp_helpers.go` (`extractOriginFromURL`), `analysis/thirdparty.go` (`extractOrigin`)

Plus `security/sri_helpers.go:46` re-implements instead of delegating.

#### M6. `os.TempDir()` / `os.MkdirTemp` still used in several test files

**Files:** `tools_generate_har_test.go`, `connection_lifecycle_helpers_test.go`, `tools_generate_warning_filter_test.go`, `internal/export/*_test.go`

These were not covered by the /tmp migration agent.

#### M7. Interact dispatch uses `interactHandlerMap()` method call vs. static map for other tools

**File:** `cmd/dev-console/tools_interact_dispatch.go`

The interact tool reconstructs its handler map on every request (lazy merge of two maps). Other tools use package-level `var xxxHandlers = map[string]Handler{...}`. Inconsistency.

### LOW Priority

#### L1. Frame error detection is dead code (TS)

**File:** `src/background/commands/analyze.ts:464,534`

`message === 'frame_not_found'` uses strict equality against a full error string like `"frame_not_found: no iframe matched..."`. The equality check never matches — frame errors always take the generic error path.

#### L2. `formatDuration` and `parseTimestamp` wrappers in capture/security are unnecessary indirection

Both are one-line delegations to `util.*`. Callers could import `util` directly.

#### L3. `getValid*Modes()` called twice in some dispatch functions

Once for `resolveToolMode` and once in the error path. Minor waste on error paths only.

#### L4. `clearBuffer` method on ToolHandler is dead code

**File:** `cmd/dev-console/tools_configure.go:74-76`

Wraps `clearConfiguredBuffer` but is never called — `toolConfigureClear` calls the impl directly.

#### L5. Test files still have 98 `@ts-nocheck` directives, 85 `time.Sleep` calls, 18+ oversized files

Significant quality debt but not blocking for release.

## Conclusion

**The codebase is release-ready.** All P0 and P1 items from the first review have been fixed and verified. The fixes were applied correctly with no regressions — full test suite passes, builds are clean, wire types are synchronized.

The remaining findings fall into two categories:
1. **Pre-existing structural debt** (Capture/ToolHandler size, test quality items) — documented, not blocking
2. **Minor issues from the fix process** (templated test headers with wrong docs links, a few missed `/tmp` paths) — low-risk, can be addressed post-release

**Recommended before release:** Fix H1 (integration test compilation) if `make test-integration` is part of the release gate. Otherwise, all other items can ship as-is with tracked follow-up work.
